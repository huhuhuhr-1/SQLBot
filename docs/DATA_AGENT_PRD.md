# Data Agent 产品需求文档（PRD）

## 1. 文档信息

- 文档名称：Data Agent 产品需求文档
- 版本：v1.0
- 状态：草案（可直接进入评审）
- 更新时间：2026-03-27
- 适用范围：SQLBot 深度分析能力升级（对话式分析 + 可追溯执行 + 报告产出）

---

## 2. 背景与问题

### 2.1 背景

现有智能分析功能可以完成问答与基础分析，但在复杂业务场景下，存在以下体验与可信度问题：

1. 用户看到较多中间步骤，难以快速抓住结论。
2. SQL 取数过程可读性弱，难以判断结论是否可信。
3. 多次查询过程缺乏统一的计划与阶段语义，回放成本高。
4. 报告输出风格不稳定，业务汇报可用性不足。

### 2.2 核心问题定义

我们需要一个“可执行、可追溯、可复用”的 Data Agent 产品形态，使用户获得：

- 一眼可读的结论报告；
- 可展开验证的证据链；
- 可复盘的执行过程；
- 可导出的交付产物（Markdown / HTML）。

---

## 3. 产品目标与范围

### 3.1 产品目标（Goals）

1. **结果导向**：默认展示结构化报告，不以中间步骤为主阅读路径。
2. **证据导向**：每条关键结论可追溯到 SQL 与结果数据。
3. **过程可控**：明确计划、阶段、执行状态与错误反馈。
4. **可复盘可导出**：支持会话回看、报告导出与复用。

### 3.2 非目标（Non-Goals）

1. 本期不做跨数据源联邦查询引擎。
2. 本期不做自动 BI 大屏编排系统。
3. 本期不做全自动数据建模（仅在现有元数据基础上分析）。

---

## 4. 目标用户与典型场景

### 4.1 目标用户

1. 业务分析师：快速回答业务问题并形成汇报材料。
2. 运营负责人：关注结论与建议，必要时下钻验证数据。
3. 数据负责人：关注口径一致性、执行可信度与复用效率。

### 4.2 典型场景

1. 销售诊断：近 30 天销售额变化、Top 客户贡献、区域差异。
2. 异常排查：订单波动、转化率下滑、渠道表现异常。
3. 周报产出：从问题到报告的一体化输出与导出。

---

## 5. 产品形态与信息架构

### 5.1 页面形态（推荐）

采用“会话区 + 报告区 + 过程详情”的组合：

1. **左侧会话区**：问题输入、会话历史、会话切换。
2. **中间主区**：步骤时间线（折叠）+ 报告卡片入口。
3. **右侧报告区**：完整报告预览、导出入口。
4. **执行详情面板（可选右侧二级）**：任务阶段、工具调用、SQL、执行状态。

### 5.2 核心对象

1. 会话（Session）
2. 分析任务（Run）
3. 执行事件（Event）
4. 报告（Report）
5. 洞察（Insight）

---

## 6. 核心业务流程

### 6.1 端到端流程

1. 用户输入问题；
2. Agent 生成计划（Plan）；
3. Agent 分阶段执行（取数、分析、沉淀）；
4. 汇总生成最终报告（Report）；
5. 持久化计划、过程、报告；
6. 用户查看、回放、导出。

### 6.2 对外阶段语义（必须）

- `plan`：任务拆解/执行计划
- `stage`：当前阶段状态
- `process`：过程细节（可折叠）
- `report`：最终报告
- `finish`：任务完成
- `error`：异常中断

---

## 7. 功能需求（Functional Requirements）

### FR-1 对话与会话管理

1. 支持新建会话、会话列表、会话切换、重命名、删除。
2. 支持会话恢复，重新打开后可看到历史报告和过程。
3. 支持在任务运行时展示进行中状态。

**验收标准**

- 用户切换会话后，历史问题与报告可正确恢复。
- 新会话不污染旧会话上下文。

### FR-2 执行计划可视化

1. 每次分析开头必须输出任务计划（复杂度、子任务、收敛规则）。
2. 计划内容可被持久化，供历史回放查看。

**验收标准**

- 每次分析均存在 `plan` 事件；
- 历史记录中可读取计划内容。

### FR-3 分阶段执行与过程追踪

1. 过程必须按 `stage/process` 事件输出。
2. 过程默认折叠，避免干扰结论阅读。
3. 同步显示关键工具动作（如取数、SQL 执行、洞察沉淀）。

**验收标准**

- 前端可按事件类型分区渲染；
- 用户可展开过程查看详细步骤。

### FR-4 报告生成（核心）

1. 最终报告必须结构化输出，至少包含：
   - 问题与口径
   - 核心结论
   - 数据支撑
   - 风险与异常
   - 建议
   - 局限
2. 报告支持 Markdown 渲染展示。
3. 报告支持导出（至少 Markdown，建议补 HTML）。

**验收标准**

- 每次分析结束前输出 `report`；
- 报告结构完整，不缺节。

### FR-5 证据链与可追溯性

1. 关键结论应可关联对应取数过程（SQL/数据摘要）。
2. 执行过程中记录工具调用状态（成功/失败/去重/空结果）。
3. 支持在详情面板查看执行日志摘要。

**验收标准**

- 至少可追溯到 SQL 文本或数据摘要；
- 出错场景可看到错误原因与阶段位置。

### FR-6 异常处理与收敛

1. 出现 SQL 错误时应回传可操作错误信息。
2. 过程支持有限重试，避免死循环。
3. 达到最大步数后强制收敛并输出已知结论与局限。

**验收标准**

- 注入 SQL 错误时，用户可看到明确错误提示；
- 超步数场景仍能正常结束并给出报告。

## 8. 非功能需求（NFR）

### NFR-1 性能

1. 首个阶段反馈时间（TTFB）目标：< 3 秒（开发环境可放宽）。
2. 报告生成完成时间：常规问题 1~5 分钟内（视数据规模）。

### NFR-2 稳定性

1. 流式链路可在前端中断后安全退出。
2. 后端异常不应导致进程崩溃，应返回 `error` 事件。

### NFR-3 可观测性

1. 记录 Run 级别日志：开始、阶段、异常、完成。
2. 保留关键事件用于问题复盘。

### NFR-4 安全与权限

1. 严格使用当前用户权限进行数据访问。
2. 禁止越权查询和跨租户数据串读。

---

## 9. 数据与事件模型（建议）

### 9.1 事件模型

统一事件协议示例：

```json
{
  "type": "stage",
  "content": "当前阶段：智能取数",
  "stage": "query",
  "chat_id": 123
}
```

### 9.2 持久化字段建议

1. `plan_markdown`
2. `process_events`（数组）
3. `report_markdown`
4. `report_html`（可选）
5. `config`（max_steps/max_data_length 等）
6. `created_at/updated_at`

---

## 10. 交互与体验要求

1. 报告优先：任务完成后用户第一视角是报告，而非过程流水。
2. 过程可见：保留折叠入口，不隐藏执行证据。
3. 状态清晰：运行中、完成、失败三态明确区分。
4. 导出便捷：报告区提供一键导出。

---

## 11. 里程碑规划

### Phase 1（MVP）

1. 打通统一事件流（plan/stage/process/report/finish/error）。
2. 完成报告结构化输出与展示。
3. 完成会话级持久化与回放。

### Phase 2（增强）

1. 增加执行详情面板（SQL/工具调用/耗时）。
2. 增加重试与重规划策略可视化。
3. 增加 HTML 报告导出。

### Phase 3（优化）

1. 提升报告模板质量与一致性。
2. 增加可配置分析模式（速度优先/深度优先）。
3. 增加质量评分与回传（报告可用性反馈）。

---

## 12. 验收标准（Release Criteria）

1. 90% 常见分析请求能产出完整六段报告。
2. 100% 分析任务可回放（至少包含计划、过程、报告）。
3. 错误场景均可见错误原因，不出现静默失败。
4. 用户可在 3 次点击内完成“查看报告 + 导出报告”。

---

## 13. 风险与应对

1. **风险：模型过程冗长影响阅读**
   - 应对：默认折叠过程，报告优先展示。
2. **风险：SQL 失败率高**
   - 应对：带错重试 + 去重 + 收敛规则。
3. **风险：报告内容不稳定**
   - 应对：固定模板 + 最终报告节点兜底生成。
4. **风险：性能波动**
   - 应对：限制步数、限制单次数据量、异步流式反馈。

---

## 14. 附录：建议任务拆分（研发）

1. 后端：事件协议与持久化改造
2. 后端：编排收敛与异常处理完善
3. 前端：报告优先布局与过程折叠
4. 前端：执行详情面板与导出能力
5. 测试：端到端场景回归（成功/失败/超时/空数据）
