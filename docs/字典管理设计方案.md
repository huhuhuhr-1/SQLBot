# SQLBot 字典管理 (Dictionary Management) 设计方案

## 一、 背景与目标

在 Text-to-SQL（ChatBI）场景中，大模型往往能准确识别表名和字段名，但对于维度字段的**枚举值（码表）**经常出现“瞎猜”现象（例如：库中存的是 `0/1`，模型生成了 `status='SUCCESS'`）。

本方案通过建立独立的**字典管理模块**，实现：
1. **语义增强**：将物理存储值（Code）映射为业务含义（Name），提升生成 SQL 的过滤条件准确率。
2. **知识复用**：一套码表可绑定到多个数据源、多个表的多个字段。
3. **智能化治理**：结合 LLM 自动推断码表含义，并监控物理数据与字典定义的偏差。

---

## 二、 模块架构

### 1. 模块定位
*   **路径**：`backend/apps/dict/`
*   **定位**：与 `terminology`（术语库）、`data_training`（SQL 示例）平级。
*   **独立性**：不放在 `settings/` 下，因为字典是核心业务元数据，涉及复杂的 CRUD、质量检测及 AI 智能化逻辑。

### 2. 核心架构图
```text
┌─────────────────┐       ┌─────────────────┐       ┌─────────────────┐
│      Dict       │       │    DictItem     │       │   DictBinding   │
│   (字典主表)    │ 1 : N │   (字典项明细)  │ N : 1 │  (字段绑定关系)  │
├─────────────────┤       ├─────────────────┤       ├─────────────────┤
│ id, name, code  │       │ id, dict_id     │       │ id, dict_id     │
│ oid, enabled    │       │ code, name      │       │ ds_id, table    │
│ description     │       │ sort, enabled   │       │ column, enabled │
└─────────────────┘       └─────────────────┘       └─────────────────┘
```

## 三、 数据模型设计 (DDL)

### 1. 字典主表 (`dict`)
存储字典的定义，如“性别”、“订单状态”。
- `id`: 主键
- `oid`: 组织/空间 ID (支持多租户隔离)
- `name`: 字典名称 (如: 订单状态)
- `code`: 字典编码 (如: ORDER_STATUS)
- `description`: 描述说明

### 2. 字典项明细 (`dict_item`)
存储具体的键值对，如 `1 -> 已支付`。
- `id`: 主键
- `dict_id`: 关联主表 ID
- `code`: 物理取值 (如: 1, SUCCESS)
- `name`: 业务显示名 (如: 已支付)
- `sort`: 排序序号
- `enabled`: 是否启用

### 3. 字典绑定关系 (`dict_binding`)
**核心复用逻辑**：通过此表将一个字典逻辑挂载到具体的物理字段上，不修改 `CoreField` 原有结构。
- `id`: 主键
- `dict_id`: 关联字典 ID
- `datasource_id`: 数据源 ID
- `table_name`: 物理表名
- `column_name`: 物理字段名
- `enabled`: 是否生效

## 四、 智能化能力 (Intelligence)

结合 SQLBot 已有的智能备注能力，字典模块应具备以下“活”的特征：

### 1. 智能识别与建议 (Smart Discovery)
*   **联动智能备注**：当用户对数据源执行「智能备注」时，LLM 若识别到字段具有枚举特征，会自动生成结构化建议。
*   **动作**：系统提示“检测到 `status` 字段为枚举类型，包含 [0:待审, 1:通过]，是否一键转为标准字典并绑定？”

### 2. 采样推断 (Sample & Infer)
*   **零录入成本**：用户选中一个字段后，点击“智能推断”。
*   **逻辑**：系统执行 `SELECT DISTINCT` 采样物理取值 -> 将 Top 取值发给 LLM -> LLM 结合上下文推断出对应的业务名称（Label） -> 自动填充 `DictItem`。

### 3. 漂移检测 (Drift Detection)
*   **数据一致性监控**：定期扫描已绑定字典的字段。
*   **发现新值**：如果物理库中出现了字典未定义的新 Code（如库里新增了状态 `4`），系统自动触发告警并建议管理员通过 AI 推断新值的含义。

## 五、 SQL 生成注入逻辑 (Prompt Strategy)

在生成 SQL 的过程中，字典信息将动态注入到 Prompt 的 Schema 描述中：

1.  **查询阶段**：`LLMService` 在构建上下文时，根据当前选中的 `tables`，查询 `dict_binding` 获取关联的字典及明细。
2.  **注入格式**：
    ```xml
    <column name="order_status">
      <type>int</type>
      <comment>订单状态</comment>
      <enums>
        <item code="0" label="待支付"/>
        <item code="1" label="已完成"/>
      </enums>
    </column>
    ```
3.  **约束**：
    - 仅注入已启用的字典项。
    - 若字典项过多（超过 20 条），仅注入 Top N 条，或提示模型优先参考关联知识。

## 六、 维护与操作流程

### 1. 字典管理流程
- **创建/导入**：支持手动创建或 Excel 批量导入。
- **质量检测**：提供“一键检测”按钮，计算字段的**覆盖率**和**命中率**。

### 2. 字段绑定流程
- **数据源视图**：在字段配置列表中，增加「字典关联」下拉框。
- **解耦操作**：修改字典内容，所有绑定的字段在下一次 AI 对话时立即生效，无需重新同步元数据。

---

## 七、 方案优势

1.  **无损集成**：不修改核心资产表 `CoreField`，对现有系统影响最小。
2.  **高度复用**：解决企业中广泛存在的“不同表、同含义”码表对齐问题。
3.  **闭环治理**：通过 AI 辅助录入与物理数据监控，确保元数据的生命力。

## 八、实现说明（当前代码库）

已实现内容：

- **后端**：`backend/apps/dict/`（模型 `biz_dict` / `biz_dict_item` / `biz_dict_binding`）、REST API 前缀 `/system/dict`，权限与术语一致（工作空间管理员）；Alembic：`067_biz_dictionary.py`（`b2c3d4e5f6g7`）、`068_biz_dictionary_ensure_tables.py`（在版本已指向 067 但表未建时幂等补建表）。
- **注入**：在 [`get_table_schema`](backend/apps/datasource/crud/datasource.py) 中按 `oid + datasource_id` 加载绑定，在对应字段的 M-Schema 行追加 `enums:code=label|...`（默认最多 20 条，超出追加 `...(truncated)`）。
- **前端**：设置侧栏「数据字典」→ `/set/dictionary`，字典维护 + 字段绑定（数据源 / 表 / 列 / 字典选择）。

未实现（仍按上文「智能化」规划，可后续迭代）：

- Excel 导入导出、采样推断、漂移检测、智能备注一键转字典。
- **助手动态数据源**（虚拟 `datasource id`）下的字典绑定与注入；当前仅对平台内 **Core 数据源** 生效。
