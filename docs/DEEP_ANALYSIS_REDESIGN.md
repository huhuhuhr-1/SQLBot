# 深度分析整体改造方案

**版本**: 1.0  
**日期**: 2026-03-15  
**依据**: 现有实现（PlanAgent、`get_data`、深度分析页）、`frontend/src/views/deep-analysis/PROTOTYPE_DESIGN.md`、产品/流程讨论结论。

---

## 1. 背景与目标

### 1.1 现状简要

| 模块 | 现状 |
|------|------|
| **Agent** | LangChain `AgentExecutor`，对外流式输出 `# 分析任务` + `# 分析过程` + `## 分析步骤 N`（N 随迭代涨）。 |
| **取数** | `get_data` → 生成 SQL（含术语 + SQL 示例）→ 真执行；工具返回整表 `df.to_json()`（上限约千行）。 |
| **前端** | 按 SSE 片段堆叠为「步骤列表 + 思考折叠」，用户看到的是长流水账。 |

### 1.2 改造目标

1. **叙事**：对外只呈现「三层流程 + 一份固定结构报告」，不再把内部「步骤 1…N」当主阅读路径。  
2. **可信**：先定主事实源与口径，少重复查同表，结论有数据支撑与局限说明。  
3. **可控**：统一取数条数、SQL 失败可带错重试、控制进入模型的 token 体量。  
4. **产品**：默认展示「综合报告」；取数/SQL 过程可收起，与 PROTOTYPE 两栏/证据链可对齐。

## 2. 现状问题清单（需对症改造）

| ID | 问题 | 优先级 |
|----|------|--------|
| P1 | `max_data_length`（请求/Prompt）与 `AnalysisContext.max_data_size`（写死 1000）不一致 | 高 |
| P2 | SQL 执行失败后，未必以 `exec-sql-err` 等形式进入下一轮 SQL Prompt，带错重试弱 | 高 |
| P3 | `get_data` 工具向 Agent 返回整段 JSON，列多/行多时 token 高 | 高 |
| P4 | 多轮 `get_data` 在 scratchpad 中累积大块 observation | 中 |
| P5 | 对外步骤流水账，与读者「先结论、再依据」的心智不符 | 高 |
| P6 | OpenAPI 中 `enable_retry` 等与 Plan 未明确接线，易误解 | 低 |
| P7 | 模板占位符 `max_lenght` 拼写 | 低 |

## 3. 流程层设计（对内多步、对外三层）

### 3.1 三层对外语义

| 层 | 做什么 | 对外呈现（建议） |
|----|--------|------------------|
| **① 取数计划** | 一句话：问什么、主表/辅表、时间口径、成功/失败含义 | 报告 §1「问题与口径」+ 可选「计划摘要」卡片 |
| **② 证据与表** | 2～4 次 SQL 为宜；同表同窗口合并维度，禁止同表重复查多次 | 报告 §3「数据支撑」+ 可选「展开取数过程」（SQL/中间表） |
| **③ 结论** | 先总后分：结论 → 数字 → 局限 → 建议 | 报告 §2～§6 |

Agent **内部**仍可多步推理；**对流式通道**：可二选一或组合——

- **A**：仍推中间块，但前端**默认折叠**为「取数过程」，主区域只锚定最终报告；或  
- **B**：中间块打 `type: process` / `type: report`，UI 分区渲染，报告区仅展示 final 或解析后的六节。

### 3.2 流程规则（写入 Prompt）

1. **先定主事实源**：首轮即明写「本题以哪张表为主答 X；Y 表仅补充 Z」，避免前后换表显得跳戏。  
2. **少查重复维度**：同表、同时间段优先一条 SQL 多维度（如 月×状态 一张汇总）。  
3. **查询次数**：建议每次分析 **get_data 2～4 次**为常态上限（失败重试不计入「常态」）。  
4. **完成条件**：最终用户可见输出**必须**包含下面 §4 的六节结构（缺一视为未完成，须补 `final_answer`）。

## 4. 输出模板（最终报告，固定六节）

以下为 **Prompt + 产品「分析报告」** 的统一结构，模型最终回复须贴近此骨架（标题可微调，章节含义不可缺）。

```markdown
## 1. 问题与口径
- **问题**：（用户原问）
- **时间范围**：
- **主数据源**：（主表；辅表及角色）
- **指标说明**：（各指标含义）

## 2. 核心结论（3～5 条，先写结论）
- …

## 3. 数据支撑（表格 + 简短解释）
| 维度 | 关键数字 | 说明 |
|------|----------|------|
| … | … | … |
（可选）1～2 段 SQL 或查询思路，不必贴满全过程。

## 4. 风险与异常
- …

## 5. 建议（可执行）
- …

## 6. 局限
- …
```

**阅读路径**：用户第一眼 §2；向下 §3；§1+§6 消除「为什么有的表空、步骤为何跳转」的疑惑。

## 5. Prompt 改造要点

| 文档/位置 | 改动方向 |
|-----------|----------|
| `backend/templates/template.yaml` → `myself.plan_prompt` | ① 禁止对用户输出「## 分析步骤 1…N」式流水账作为主内容；② 要求必须先陈述主表与口径再取数；③ 要求同表少次、多维度合并；④ **最终回答必须包含 §1～§6**；⑤ 修正 `max_lenght` → `max_length`（与各处 format 一并改）。 |
| `PlanAgent.analysis()`（可选） | 仍可向队列推「过程」供调试；或改为仅推 `type: report` 的最终块 + 可选 `type: process`；**产品默认不展开 process**。 |
| `FinalAnswerTool` | description 中嵌入上述六节 Markdown 骨架，要求 `answer` 严格按节输出。 |
| `get_data` 工具描述 | 强化「同指标同窗口一条 SQL 多维度」「单轮分析 get_data 次数建议上限」。 |

（术语、SQL 示例：**已在查数链路中使用**，保持不变即可。）

---

## 6. 后端改造要点

| 项 | 说明 |
|----|------|
| **统一条数** | `OpenChatQuestion.max_data_length` → 初始化 `AnalysisContext.max_data_size`；`get_data` 拼 query 与 `plan_prompt` 中 `{max_length}` 一致。 |
| **SQL 失败带错** | `get_data_for_plan` 执行失败时，落库/会话中写入与聊天一致的 `exec-sql-err`，保证下一次 `generate_sql_for_plan` 的 `error_msg` 含真实库错；或当次在 messages 中追加 Human 错误片段。 |
| **工具返回瘦身（控 token）** | 方案择优：返回「前 K 行样本 + 行数 + 列名 + 可选摘要」+ 服务端保留全量 DF 供 `insight_analysis`；或限制 JSON 行列/字符上限。需评估与 `InsightTool`（需 df JSON）的兼容性。 |
| **完成态** | SSE 可增加 `type: report_done`，正文为六节 Markdown，便于前端只展示报告区与导出。 |

## 7. 前端改造要点（对齐 PROTOTYPE_DESIGN）

| 项 | 说明 |
|----|------|
| **默认视图** | 主区域优先展示「综合报告」（解析 `final_answer` 或整条六节 Markdown）；分析中可骨架占位 + 加载态。 |
| **过程区** | 「取数 / SQL / 中间步骤」收入 **折叠面板** 或右侧「查询链时间线」，默认收起。 |
| **i18n** | 新增：综合报告、展开取数过程、问题与口径等（中/英）。 |
| **导出** | 导出内容 = 六节报告 + 可选附录（SQL 列表）；见 PROTOTYPE「导出报告」。 |
| **双栏** | 按 PROTOTYPE：左 70% 报告+可折过程，右 30% 时间线/证据链（可分期：第一期单栏+折叠亦可）。 |

参考实现文件：`frontend/src/views/deep-analysis/index.vue`、`PROTOTYPE_DESIGN.md`。

## 8. 实施阶段建议

| 阶段 | 内容 | 产出 |
|------|------|------|
| **Phase 0** | Prompt：六节模板 + 主表优先 + 禁止步骤流水账 + 统一 max_length | 行为可先于 UI 改善 |
| **Phase 1** | 后端：max_data 对齐；SQL 失败带错；可选工具返回瘦身 PoC | 稳定性 + token |
| **Phase 2** | 前端：报告优先 + 过程折叠；SSE `type` 区分 report/process（若后端已加） | 体验对齐目标 |
| **Phase 3** | PROTOTYPE 双栏、证据链、导出 | 与原型一致 |

---

## 9. 验收标准（建议）

1. 任意典型问题跑完，**首屏或主区域**以六节报告为主，步骤列表不默认占满屏。  
2. 报告中 **§1 含主数据源与时间口径**；**§2 为 bullet 结论**；**§6 含局限**。  
3. 同一次分析，**同表同时间窗**无多次无意义的重复维度查询（可通过 SQL 日志抽查）。  
4. `max_data_length` 与单次取数上限、Prompt 描述 **数值一致**。  
5. 人为制造 SQL 错误后，**下一次生成 SQL** 的 Prompt 中含可操作错误信息（抽样日志）。
